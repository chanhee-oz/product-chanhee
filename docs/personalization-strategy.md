# 개인화 연결 전략

**풍수 진단 도구 x 오늘의집 개인화 연결 방안**

## 1. 현재 한계점

풍수 진단 도구는 위치(GPS/동네), 집 형태, 층수, 방향, 동거인 총 5가지 주거 데이터를 수집합니다. 그러나 현재 구조에는 세 가지 한계가 있습니다.

- 데이터가 브라우저 메모리에만 존재. 페이지를 닫으면 소멸합니다.
- 오늘의집 계정과 연결이 없어서 "이 사람이 강남 아파트 고층 남향에 산다"는 정보를 플랫폼에서 활용할 방법이 없습니다.
- 상품 추천 링크(store.ohou.se)를 클릭해도 어떤 풍수 유형의 유저가 클릭했는지 추적이 불가합니다.

즉, 재미있는 데이터를 모으는 데는 성공했지만, 그 데이터가 오늘의집 경험 개선으로 흘러가는 파이프가 없는 상태입니다.

## 2. 연결 방안

### 방안 A: 딥링크 + URL 파라미터 전달

결과 화면에서 오늘의집으로 이동하는 링크에 주거 데이터를 URL 파라미터로 붙이는 방식입니다.
예: `ohou.se/search?query=무드등&fs_type=sunrise&fs_house=apartment&fs_dir=east`

- **장점**: 서버 변경 0. 풍수 랜딩에서 코드 몇 줄 수정으로 즉시 가능
- **단점**: 일회성 전달. 세션 종료 시 소멸. 파라미터가 길어지면 공유 시 지저분해짐
- **구현 복잡도**: 낮음 (프론트 1일)
- **데이터 정확도**: 높음
- **프라이버시 리스크**: 낮음 (GPS 좌표 노출 주의, 동네명 수준으로 제한)

### 방안 B: 결과 기반 맞춤 큐레이션 페이지 유도

풍수 유형별로 오늘의집 내 큐레이션 페이지를 미리 만들어놓고 연결하는 방식.

- **장점**: 전환율 높을 가능성. "내 집에 맞는 추천"이라는 명확한 가치. 기존 기획전 인프라 활용
- **단점**: 12개 유형별 큐레이션 페이지를 수동 제작/관리
- **구현 복잡도**: 낮음 (콘텐츠 제작 공수가 주된 비용)
- **프라이버시 리스크**: 없음

### 방안 C: 쿠키/localStorage 기반 크로스세션 유지

풍수 결과를 localStorage에 저장하고, 오늘의집 도메인에서 읽어가는 방식. 단, 도메인이 다르므로 ohou.se 서브도메인(예: fengshui.ohou.se)으로 이관 필요.

- **장점**: 로그인 없이도 재방문 시 데이터 유지
- **단점**: 도메인 통합 필수. 앱에서는 사용 불가. 쿠키 삭제 시 소멸
- **구현 복잡도**: 중간 (도메인 이관 + 오늘의집 웹 프론트에서 localStorage 읽기)
- **프라이버시 리스크**: 중간 (동의 배너 필요)

### 방안 D: 오늘의집 로그인 연동 (OAuth) + 프로필 저장

결과 화면에서 "결과를 오늘의집에 저장하기" → OAuth 로그인 → 서버에 주거 데이터 저장.

- **장점**: 가장 완전한 연결. 앱/웹 모두 활용 가능
- **단점**: OAuth + 서버 API + 프로필 스키마 확장 필요. 가벼운 재미 콘텐츠에서 로그인 요구 시 이탈 우려
- **구현 복잡도**: 높음 (3~4주)
- **프라이버시 리스크**: 낮음 (로그인 기반이므로 동의 플로우 명확)

## 3. 비교표

| 기준 | A. URL 파라미터 | B. 큐레이션 페이지 | C. localStorage | D. OAuth 연동 |
|---|---|---|---|---|
| 구현 복잡도 | 낮음 (1일) | 낮음 (콘텐츠 공수) | 중간 (1~2주) | 높음 (3~4주) |
| 데이터 지속성 | 일회성 | 없음 | 브라우저 한정 | 영구 |
| 앱 활용 | 딥링크 가능 | 가능 | 불가 | 완전 |
| 유저 마찰 | 없음 | 없음 | 없음 | 로그인 필요 |
| 추천 정밀도 | 중간 | 유형 수준 | 높음 | 최고 |
| 프라이버시 | 낮은 리스크 | 없음 | 동의 필요 | 명확 |

## 4. 추천 로드맵

### Phase 1 — 지금 당장 (서버 변경 없이, 1~2일)

A + B 조합. 상품 링크에 UTM + 풍수 파라미터 추가. 동시에 인기 유형에 대해 오늘의집 검색 URL로 직접 연결.
이것만으로도 "풍수에서 넘어온 유저"의 행동을 GA에서 세그먼트로 분리 가능.

### Phase 2 — 바이럴 검증 후 (2~4주)

도메인을 ohou.se 서브도메인으로 이관 + 방안 C 적용. 비로그인 유저도 재방문 시 "지난번 진단 결과" 표시 가능. 오늘의집 웹 메인에서 주거 맥락 기반 추천 실험 시작.

### Phase 3 — PMF 확인 후 (1~2개월)

방안 D를 결과 저장 옵션으로 추가. 강제가 아니라 "오늘의집에 저장하면 맞춤 추천을 받을 수 있어요" 형태. 주거 데이터를 유저 프로필 속성으로 통합하여 검색 랭킹, 피드 큐레이션, 푸시 개인화에 활용.

## 5. 데이터 활용 시나리오

| 수집 데이터 | 활용 예시 |
|---|---|
| 위치(동네) | 배송 가능 상품 우선 노출, 지역별 인기 상품, 인근 시공 업체 |
| 집 형태 | 원룸→공간 활용 가구, 아파트→베란다 상품, 단독→정원 카테고리 |
| 층수 | 고층→커튼/블라인드(조망), 저층→방범/프라이버시 상품 |
| 방향 | 서향→암막커튼/UV 필름, 북향→조명/간접등 가중치 |
| 동거인 | 반려동물→펫 프렌들리, 가족→안전 가구/키즈, 혼자→1인 큐레이션 |

이 조합을 활용하면 "서향 원룸에 혼자 사는 유저"와 "남향 아파트에 가족과 사는 유저"에게 완전히 다른 홈피드를 보여줄 수 있습니다. 현재 오늘의집이 보유하지 못한 주거 맥락 데이터이며, 이것이 이 프로젝트의 데이터 가치입니다.
