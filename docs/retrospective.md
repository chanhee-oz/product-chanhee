# 풍수 진단 도구 — 프로젝트 회고

## 프로젝트 개요

오늘의집 고객 대상 인터랙티브 풍수 진단 도구. 풍수를 재미 요소로 활용해 주거 데이터(위치, 집 형태, 층수, 방향, 동거인)를 자연스럽게 수집하는 바이럴 프로토타입.

**목적**: 고객이 재미있게 참여하면서 주거 관련 데이터를 확보 → 향후 탐색/추천 경험 개선에 활용 가능성 검증.

## 개발 타임라인

### Phase 1: MVP (2026-04-02 ~ 04-07)

**만든 것:**
- 인트로 → Q1~Q5 → 로딩 → 결과 전체 플로우
- 12개 풍수 유형 체계 + 매핑 로직 (방향 × 층수 + 집형태/동거인 override)
- GPS/검색 기반 위치 입력
- 에너지 리딩 (활력/안정/풍요 바 그래프)
- 결과 이미지 저장 (html2canvas) + 링크 공유 (URL 파라미터)
- 공유 링크 진입 → "나도 해보기" 전환 플로우
- 피드백 수집 (Google Form 백그라운드 전송)

**핵심 결정:**
- Vanilla JS + 정적 호스팅. 프레임워크 도입하지 않음 → 빠른 프로토타이핑, 간단한 배포
- 서버 없이 클라이언트만으로 동작 → 프라이버시 보장, 개발 속도 극대화
- 상품 추천은 데모 수준(2개 공통 상품)으로 시작

**잘 된 것:**
- 전체 플로우가 30초 내 완료 가능 → 이탈 방지에 유리
- 풍수라는 컨셉이 흥미 유발에 효과적 (테스트 유저 반응 긍정적)
- 피드백 수집 체계를 MVP에 포함시킨 것 → 데이터 기반 개선 가능

**아쉬웠던 것:**
- 상품 추천이 데모 수준에서 멈춤 (모든 유형에 같은 상품 2개)
- 카카오톡 SDK 미연동 → 바이럴 핵심 채널이 작동하지 않음
- 행동 데이터 트래킹 없음 → 어디서 이탈하는지 파악 불가
- 오늘의집 서비스와의 연결 없음 → 수집한 데이터를 활용할 경로 없음

### Phase 2: 분석 체계 + 상품 확장 + 전략 (2026-04-12)

**만든 것:**

1. **상품 추천 확대**
   - 12유형별 맥락에 맞는 상품 2개씩 매핑 (총 24개)
   - 오늘의집 검색 URL 기반 → 항상 최신 상품 노출, 품절/이미지 깨짐 없음
   - 모든 링크에 UTM 파라미터 포함 → 유형별 유입/전환 추적 가능
   - companion 기반 보조 추천: pet → 반려동물 인테리어, family → 키즈 인테리어

2. **행동 데이터 트래킹**
   - Vercel Web Analytics (쿠키 없음, 프라이버시 친화) + PostHog + GA4 3중 구조
   - analytics.js 모듈: 10개 커스텀 이벤트 (screen_view, screen_exit, option_selected, location_method, result_viewed, product_clicked, share_action, feedback_submitted, session_end, shared_link_view)
   - 이탈 감지: visibilitychange/beforeunload로 마지막 화면 + 최고 도달 지점 기록
   - 상품 클릭 자동 추적: event delegation으로 동적 생성 카드도 커버

3. **개인화 전략 문서**
   - 4가지 연결 방안 비교: URL 파라미터 / 큐레이션 페이지 / localStorage / OAuth
   - Phase 1~3 로드맵: 지금 당장(UTM) → 도메인 이관 → OAuth 통합
   - 수집 데이터별 활용 시나리오 (위치→배송, 집형태→카테고리, 방향→상품, 동거인→필터)

4. **피드백 분석 워크플로우**
   - 분석 프레임워크: 세그멘테이션, 만족도(match rate), CTR/CVR, 공유 행동
   - 주간 자동 리포트: scheduled task (매주 월요일 09:00 실행)
   - 이상 징후 알림 조건 정의 (match rate 급락, mismatch 급증 등)
   - BRD 문서화 구조 설계

5. **매핑 로직 개선** (유저 시뮬레이션 기반)
   - apartment_family → mountain, villa_family → fertile override 추가
   - 에너지 최솟값 50으로 상향 (부정 인상 방지)
   - Q5 "아이와 함께" → "가족과 함께" 문구 변경
   - 카카오톡 SDK Kakao.init() 활성화 (JS키 교체 필요)

**핵심 결정:**
- GA4 대신 Vercel Analytics를 우선 채택 → 쿠키 없음, GDPR 걱정 없음, Vercel 배포 환경에 네이티브
- 상품은 개별 상품 ID 대신 검색 URL로 연결 → 유지보수 비용 0, 항상 최신
- 남향+고층+가족(한국 최다 주거 조합) 매핑을 fire → mountain으로 변경 → 예상 만족도 대폭 개선

**방법론:**
- 유저 시뮬레이션(6개 페르소나)으로 실제 경험 검증 후 개선 → 가설이 아닌 시나리오 기반 의사결정
- 코드 리뷰를 통해 Critical 2개 + Important 4개 이슈 발견 및 수정

## 배운 것

### 프로덕트
- **가장 흔한 조합이 가장 중요하다.** 남향+아파트+가족은 전체 유저의 상당수를 차지하는데, 초기 매핑이 이 조합에 직관적이지 않았음. 데이터 분포를 먼저 고려했어야 함.
- **"결과는 항상 긍정적"이라는 원칙은 숫자에도 적용되어야 한다.** 텍스트는 긍정적이었지만 에너지 40%라는 숫자 자체가 부정 인상을 줌.
- **상품 추천은 유형만으로 부족하다.** companion 데이터(pet/family)를 상품 추천에 활용하니 훨씬 자연스러워짐. 수집하는 데이터를 최대한 활용해야 함.

### 기술
- **검색 URL > 개별 상품 URL.** 개별 상품 ID는 품절/가격 변동/이미지 깨짐 리스크가 있지만, 검색 URL은 항상 최신 결과를 보여줌. 유지보수 비용 차이가 큼.
- **Analytics는 처음부터 넣었어야 함.** MVP에 트래킹 없이 런칭하면 "잘 되고 있는지" 판단할 근거가 없음. 다음 프로토타입에는 Day 1부터 기본 이벤트 트래킹 포함.
- **Vercel Analytics는 정적 사이트에 적합.** 쿠키 없고, 스크립트 1줄이면 끝. GA4보다 설정이 훨씬 간단.

### 프로세스
- **유저 시뮬레이션이 효과적이었다.** 6개 페르소나로 전체 플로우를 돌려보니 실제 데이터 없이도 매핑 약점, 상품 미스매치, UX 마찰점을 발견할 수 있었음. 코드 리뷰보다 더 많은 인사이트를 줌.
- **Hook이 편집을 중단시키는 이슈.** `ohou.se/images/...` URL 때문에 edit hook이 매번 작업을 중단시킴. 원인 파악에 시간이 걸렸음. Hook의 동작 원리를 미리 파악했으면 좋았을 것.
