# PRD: 풍수 진단 도구 — 주거 데이터 수집 바이럴 프로토타입

**문서 소유자**: Chanhee Lee (CPO)
**최종 업데이트**: 2026-04-12
**상태**: 프로토타입 운영 중 (Phase 2 완료)

---

## 1. 배경

오늘의집은 인테리어/홈 커머스 플랫폼이지만, 고객이 어떤 집에 사는지에 대한 구조화된 데이터를 거의 보유하지 못하고 있다. 검색/추천 알고리즘은 행동 데이터(클릭, 구매)에 의존하는데, "남향 아파트 20층에 가족과 사는 사람"과 "북향 원룸에 혼자 사는 사람"이 같은 검색 결과를 본다.

주거 맥락 데이터(위치, 집 형태, 층수, 방향, 동거인)를 확보하면:
- 검색 결과에 주거 맥락 가중치 적용 가능 (서향 → 암막커튼 상위 노출)
- 추천 피드 개인화 (원룸 → 공간 활용 가구, 반려동물 → 펫 인테리어)
- 지역 기반 서비스 연결 (배송, 시공, 인근 매장)

문제는 이 데이터를 어떻게 수집하느냐이다. 직접적인 설문("집 정보를 입력해주세요")은 참여율이 극히 낮다. 재미가 없기 때문이다.

## 2. 목적

**풍수 진단이라는 엔터테인먼트 포맷으로 주거 데이터를 자연스럽게 수집하는 바이럴 프로토타입을 만들어 검증한다.**

검증하려는 가설 3가지:
1. 풍수라는 컨셉으로 포장하면 주거 정보 입력의 심리적 저항이 낮아진다
2. 결과를 공유하고 싶은 콘텐츠로 만들면 자연 유입(바이럴)이 발생한다
3. 수집한 주거 데이터가 실제로 추천/탐색 경험 개선에 의미 있는 신호가 된다

## 3. 기대 성과

### 정량 (런칭 후 4주 기준)
- 퍼널 완주율: 70% 이상 (인트로 → 결과 도달)
- 피드백 Match Rate: 70% 이상 ("맞는 것 같아요" 비율)
- 공유율: 결과 도달 유저 중 20% 이상이 1가지 이상 공유 행동
- 상품 CTR: 결과 도달 유저 중 10% 이상이 상품 카드 클릭

### 정성
- 수집 데이터(집 형태, 방향, 층수, 동거인)의 분포가 현실적인지 확인 → 데이터 신뢰도 검증
- 공유 링크를 통한 2차 유입이 자연 발생하는지 확인 → 바이럴 루프 작동 검증
- 수집한 주거 맥락으로 상품 추천 시나리오가 실제로 유의미한지 판단할 수 있는 초기 데이터 확보

## 4. 유저 스토리 / 주요 유즈 케이스

### UC1: 직접 참여
SNS/메신저에서 링크를 보고 들어온 고객이, 탭 5번으로 풍수 진단을 완료하고, 결과("떠오르는 태양의 집")를 받고, 추천 상품을 탐색하거나 친구에게 공유한다.

### UC2: 공유 링크 진입
친구의 결과("타오르는 불꽃의 집")를 보고 호기심이 생겨, "나도 해보기" 버튼을 눌러 직접 진단을 시작한다.

### UC3: 결과 불일치 피드백
결과가 본인 집 느낌과 다르다고 느낀 고객이, "좀 다른 느낌이에요" → 구체적 사유("더 차분한 느낌")를 선택한다. 이 데이터가 매핑 로직 개선에 사용된다.

### UC4: 상품 탐색 전환
결과 화면의 팁("현관에 작은 초록 식물을 두면...") 아래 연결된 상품("미니 그린 식물 화분")을 탭해 오늘의집 검색 페이지로 진입, 실제 상품을 탐색/구매한다.

## 5. 작동 방식

### 플로우
```
인트로 → Q1(위치: GPS/검색) → Q2(집 형태) → Q3(층수) → Q4(방향) → Q5(동거인, 복수선택) → 로딩 연출 → 결과
```

### 결과 매핑 로직
- **1차 매핑**: 방향(5) × 층수(5) = 25개 조합 → 12개 풍수 유형 중 하나로 매핑 (`TYPE_MAPPING`)
- **2차 Override**: 특정 집형태+동거인 조합은 1차 결과를 덮어씀 (`HOUSE_OVERRIDES`)
  - 예: 아파트+가족 → "단단한 산의 집" (안정 강조), 원룸+혼자 → "고요한 달의 집"
- **에너지 보정**: 동거인별로 활력/안정/풍요에 ±5~10 보정. 최솟값 50, 최댓값 100
- **보조 상품**: pet 선택 시 반려동물 인테리어, family 선택 시 키즈 인테리어 자동 추가

### 12가지 풍수 유형
| 유형 | 키워드 | 대표 조건 |
|---|---|---|
| 떠오르는 태양의 집 | 활력, 시작, 성장 | 동향 + 중층 |
| 흐르는 물의 집 | 유연함, 풍요, 소통 | 방향 미상 + 중층 |
| 단단한 산의 집 | 안정, 신뢰, 보호 | 아파트/단독주택 + 가족 |
| 고요한 달의 집 | 평온, 직관, 회복 | 북향 + 고층 또는 원룸+혼자 |
| 타오르는 불꽃의 집 | 열정, 창의, 에너지 | 남향 + 중/고층 (가족 없는 경우) |
| 깊은 숲의 집 | 치유, 성찰, 재생 | 단독주택 + 반려동물 |
| 바람길의 집 | 변화, 자유, 모험 | 동향 고층 또는 서향 초고층 |
| 옥토의 집 | 풍요, 결실, 나눔 | 서향 중층 또는 빌라+가족 |
| 별빛의 집 | 영감, 꿈, 가능성 | 초고층 (동/서/남향) |
| 새벽안개의 집 | 신비, 직관, 잠재력 | 북향 + 저/중층 |
| 돌다리의 집 | 견고, 인내, 성실 | 서향 저층 또는 방향 미상 저층 |
| 봄바람의 집 | 새로움, 희망, 설렘 | 동향 + 저층 |

### 결과 화면 구성
1. 유형 타이틀 + 한 줄 요약 + 키워드 태그
2. 풍수 해설 ("어떻게 읽었냐면요...") — 입력 항목별 풍수 근거
3. 에너지 리딩 — 활력/안정/풍요 바 그래프 (최솟값 50%)
4. 개선 팁 + 연결 상품 추천 (오늘의집 검색 URL, UTM 포함)
5. 보조 상품 추천 (pet/family 기반, 해당 시에만)
6. 공유 (카카오톡/X/링크복사/이미지 저장)
7. 피드백 ("이 결과, 우리 집이랑 어울려요?")

### 공유 & 바이럴 루프
```
결과 화면 → 공유 (카카오/X/링크/이미지)
                ↓
        친구가 ?type=xxx 링크로 진입
                ↓
        친구 결과 미리보기 + "나도 해보기" CTA
                ↓
        새로운 유저가 처음부터 진단 시작
```

## 6. In-Scope

- 5단계 질문 플로우 (위치/집형태/층수/방향/동거인)
- 12가지 풍수 유형 결과 + 풍수 해설
- 유형별 맥락 맞는 상품 추천 (오늘의집 검색 URL 기반)
- companion 기반 보조 상품 추천 (pet/family)
- 결과 이미지 저장 + 카카오/X/링크 공유
- 공유 링크 진입 → "나도 해보기" 바이럴 루프
- 피드백 수집 (Google Form 백그라운드 전송)
- 행동 데이터 트래킹 (Vercel Analytics + PostHog + GA4)
- 주간 자동 분석 리포트 (scheduled task)
- 모바일 퍼스트 반응형 (max-width 480px, 데스크톱 카드 레이아웃)
- 오늘의집 가입/로그인 CTA (결과 화면, "5,000원 할인 쿠폰 받기")
- 풍수 데이터 핸드오프 구조 (URL 파라미터 `fs_data` + localStorage 이중 저장)

## 7. Out-of-Scope

- 오늘의집 서버에서 `fs_data`를 수신/저장하는 로직 (내부 개발팀 협업 필요)
- 실제 쿠폰 발급 연동 (마케팅팀 협업 필요)
- 유저 데이터 서버 저장 (현재 브라우저 + URL 파라미터로만 전달)
- 지도에서 직접 위치 찍기
- 이사 좋은 날, 계절별 풍수 운세 등 추가 콘텐츠
- A/B 테스트 인프라
- 프레임워크 도입 (React, Vue 등)

## 8. 기술 제약 / 설계 원칙

- **Vanilla JS + 정적 호스팅.** 빌드 스텝 없음. CDN 의존성 최소 (html2canvas, Kakao SDK)
- **서버 전송 없이 클라이언트만 동작.** 위치 포함 모든 응답은 브라우저에서만 처리
- **상품 추천은 검색 URL.** 개별 상품 ID가 아닌 `ohou.se/search/index?query=...` → 항상 최신 상품, 유지보수 0
- **결과는 항상 긍정적.** 부정적 진단 없음. 에너지 수치 최솟값 50%
- **재미 > 정확성.** 풍수의 학술적 정확도보다 엔터테인먼트 가치 우선
- **프라이버시 우선.** Analytics는 쿠키 없는 Vercel Analytics 우선. 위치 정보는 구글폼에 전송하지 않음

## 9. 데이터 수집 & 활용 전략

### 수집하는 데이터
| 데이터 | 수집 방법 | 저장 위치 |
|---|---|---|
| 위치 (동네) | GPS 또는 검색 | 브라우저 메모리만 (미전송) |
| 집 형태 | 5지선다 | 피드백 시 Google Form에 조합으로 전송 |
| 층수 | 5지선다 | 〃 |
| 방향 | 5지선다 | 〃 |
| 동거인 | 복수선택 | 〃 |
| 결과 유형 | 자동 계산 | Google Form 전송 |
| 만족도 | match/mismatch 클릭 | Google Form 전송 |
| 행동 이벤트 | 자동 수집 | Vercel Analytics + PostHog + GA4 |

### 활용 시나리오 (Phase 3 이후)
| 수집 데이터 | 개인화 적용 |
|---|---|
| 위치 | 배송 가능 상품 우선 노출, 지역별 인기 상품, 인근 시공 업체 |
| 집 형태 | 원룸→공간 활용, 아파트→베란다, 단독→정원 카테고리 가중치 |
| 층수 | 고층→커튼/블라인드, 저층→방범/프라이버시 상품 |
| 방향 | 서향→암막커튼, 북향→조명/간접등 |
| 동거인 | pet→펫 프렌들리, 가족→안전 가구/키즈, 혼자→1인 큐레이션 |

### 개인화 연결 로드맵
- **Phase 1 (현재)**: 상품 링크에 UTM 포함. GA에서 fengshui 소스 세그먼트 분리 가능
- **Phase 2**: ohou.se 서브도메인 이관 + localStorage로 크로스세션 유지
- **Phase 3**: OAuth 연동 + 유저 프로필에 주거 데이터 저장 → 검색/피드 개인화

## 10. 변경 이력

### 2026-04-02: 초기 설계 및 MVP 구현 시작

**결정**: Vanilla JS + 정적 호스팅으로 프로토타입 구축
**이유**: 빠른 검증이 목적. 프레임워크 없이 HTML/CSS/JS만으로 하루 안에 동작하는 프로토타입을 만들 수 있음. 배포도 GitHub Pages / Vercel로 즉시 가능.

구현 순서: HTML 구조 → 질문 카드 UI → GPS/검색 → 12개 유형 데이터 + 매핑 → 결과 렌더링 → 이미지/URL 공유 → 반응형 + OG 메타

### 2026-04-02 ~ 04-07: MVP 완성 + 피드백 기능

**추가**: 공유 버튼 카카오/X/링크 3종, 상품 추천(데모 2개), 피드백 수집(Google Form)
**결정**: 피드백을 Google Form pre-filled URL로 백그라운드 전송
**이유**: 서버 없이 데이터 수집하는 가장 간단한 방법. 고객은 폼을 보지 않고 클릭만으로 피드백 제출. 응답은 Google Sheets에 자동 적재되어 분석 가능.

**UI 개선**:
- 화면 전환을 slide → fade+scale로 변경 (PC에서 프레임 밖으로 벗어나는 문제)
- 인트로에 소셜프루프 추가 ("50,000명이 해봤어요")
- 단독주택 선택 시 층수 자동 설정 + 확인 화면 (불필요한 질문 스킵)
- Q1 GPS 버튼을 CTA 스타일로 강조

### 2026-04-12: 분석 체계 + 상품 확대 + 매핑 개선

**추가 1: 상품 추천 확대**
- 기존: 12개 유형 전부 동일한 데모 상품 2개
- 변경: 유형별 맥락 맞는 상품 2개씩 + pet/family 보조 상품
- **결정**: 개별 상품 ID 대신 오늘의집 검색 URL 사용
- **이유**: 개별 상품은 품절/가격변동/이미지 깨짐 리스크. 검색 URL은 항상 최신 상품을 보여주고 유지보수 비용이 0.

**추가 2: 행동 데이터 트래킹**
- Vercel Web Analytics + PostHog + GA4 3중 구조
- 10개 커스텀 이벤트: screen_view, screen_exit, option_selected, location_method, result_viewed, product_clicked, share_action, feedback_submitted, session_end, shared_link_view
- **결정**: GA4 단독이 아닌 Vercel Analytics 우선 + 복수 도구 병행
- **이유**: Vercel Analytics는 쿠키 없음(GDPR 걱정 없음), Vercel 배포 환경에 네이티브. PostHog은 세션 리플레이/퍼널 분석에 강점. GA4는 UTM/코호트/Google Ads 연동에 필수. 각각 강점이 달라 병행.

**추가 3: 매핑 로직 개선 (유저 시뮬레이션 기반)**
- 6개 페르소나로 전체 플로우 시뮬레이션 → 문제 발견 → 수정
- **핵심 변경**: `apartment_family → mountain`, `villa_family → fertile` override 추가
- **이유**: 남향+고층+아파트+가족은 한국에서 가장 흔한 주거 조합인데, 기존 매핑에서 "타오르는 불꽃의 집"(안정 45%)이 나옴. 가족이 있는 유저에게 안정 에너지가 낮게 나오면 결과 만족도가 급락. override로 "단단한 산의 집"(안정 100%)으로 변경.
- **에너지 최솟값 50% 상향**: `Math.max(0,...)` → `Math.max(50,...)`. 어떤 수치든 40% 이하가 나오면 "우리 집이 별로인가?" 부정 인상을 줌. 긍정적 프레이밍 원칙에 맞게 하한선 설정.
- **Q5 문구**: "아이와 함께 북적북적" → "가족과 함께 북적북적". 성인자녀, 부모님과 동거하는 케이스를 포괄하지 못했음.

**추가 4: 전략 문서 + 자동화**
- 개인화 연결 전략 (4가지 방안 비교 + Phase 1~3 로드맵)
- 피드백 분석 워크플로우 (세그멘테이션/만족도/CTR 프레임워크)
- 주간 자동 분석 리포트 scheduled task (매주 월요일 09:00)

**추가 5: 오늘의집 가입/로그인 CTA + 데이터 핸드오프**
- 결과 화면에 "오늘의집에서 우리 집 꾸미기 — 5,000원 할인 쿠폰" CTA 추가
- **데이터 전달 구조** (이중):
  1. URL 파라미터 `fs_data` — 풍수 응답 전체를 base64 인코딩하여 가입/로그인 URL에 포함. 오늘의집 서버에서 디코딩 → 유저 프로필 저장 가능
  2. localStorage `fengshui_data` — 같은 데이터를 로컬에도 저장. Phase 2(도메인 이관) 후 크로스세션 활용
- **결정**: 실제 서버 수신/저장은 Out-of-Scope (내부 개발팀 협업 필요). 프론트에서 데이터를 준비하고 전달하는 구조만 완성
- **이유**: 현재 가장 큰 한계가 "수집한 데이터를 고객 단위로 저장할 방법이 없음". 서버 연동 전이라도 CTA를 노출하면 (1) 가입 전환 측정 가능, (2) 서버 연동 시 바로 데이터 수신 가능, (3) UTM으로 풍수→가입 경로 추적 가능

**`fs_data` 페이로드 구조** (base64 디코딩 시):
```json
{
  "fs_type": "mountain",
  "fs_direction": "south",
  "fs_floor": "high",
  "fs_house": "apartment",
  "fs_companion": "partner,family",
  "fs_location": "서울 강남구 역삼동",
  "fs_timestamp": "2026-04-12T15:30:00.000Z"
}
```

**오늘의집 내부 개발팀이 해야 할 것** (Phase 3):
1. 가입/로그인 URL의 `fs_data` 파라미터를 수신하는 핸들러 추가
2. base64 디코딩 → JSON 파싱 → 유저 프로필에 주거 데이터 저장
3. 풍수 진단 완료 유저에게 실제 쿠폰 발급 로직 연동
4. 저장된 주거 데이터를 검색/추천/피드 개인화에 활용하는 신호로 연결

## 12. 성공 기준 및 다음 단계

### 프로토타입 성공 기준 (런칭 4주 후 판단)
1. 퍼널 완주율 70%+ → "콘텐츠가 재미있다"
2. Match Rate 70%+ → "결과가 그럴듯하다"
3. 공유율 20%+ → "바이럴 가능성 있다"
4. 공유 링크 → 신규 유저 전환율 50%+ → "바이럴 루프가 작동한다"

### 성공 시 다음 단계
- Phase 2: ohou.se 서브도메인 이관, localStorage 활용, 마케팅 채널 투입
- Phase 3: OAuth 연동, 유저 프로필에 주거 데이터 저장, 플랫폼 전체 개인화 적용

### 실패 시 학습
- 퍼널 완주율 낮음 → 질문이 너무 많거나 재미 없음. 3문항으로 축소 테스트
- Match Rate 낮음 → 매핑 로직 재설계 또는 유형 수 축소
- 공유율 낮음 → 결과 콘텐츠의 공유 가치 부족. 결과 이미지/네이밍 개선
- 전체적으로 관심 없음 → 풍수 컨셉 자체를 재고. 다른 엔터테인먼트 포맷 시도
